## 1. 推荐的第二遍源码阅读顺序

第一遍通过教程建立全景；第二遍按依赖由少到多阅读：


本章普通 Python 实验默认已按第 2 章执行 `python -m pip install -e <项目目录>`，或已把项目的 `src` 目录加入 `PYTHONPATH`。

### 2.2 交互实验

在项目目录执行 Python：

```python
from joint_position_recorder.motion_planner import ConstantSpeedPlanner

planner = ConstantSpeedPlanner(arrival_tolerance_degrees=0.01)
current = (0.0, 0.0, 0.0, 0.0)
target = (1.0, 2.0, -1.0, 0.5)
speed = (2.0, 4.0, 1.0, 1.0)

for frame in range(25):
    result = planner.step(current, target, speed, dt=0.05)
    current = result.positions_degrees
    print(frame, current, result.reached)
    if result.all_reached:
        break
```

观察每个关节的步长、到达帧数，以及已到达关节如何一直保持目标。

## 5. 第四阶段：用 FakeAdapter 驱动 Controller

### 5.1 最小 Adapter

```python
class MemoryAdapter:
    def __init__(self, positions):
        self.ready = True
        self.positions = tuple(positions)

    def get_positions_degrees(self):
        return self.positions

    def set_positions_degrees(self, values):
        self.positions = tuple(values)

    def hold_current_position(self):
        return self.positions
```

### 5.2 组装 Controller

```python
from pathlib import Path

from joint_position_recorder.config import load_project_config
from joint_position_recorder.controller import MotionController

project = Path("D:/learning/IntelligentDepartment/CodesSet/Self/260707IsaacSIm/scripts/260720_01JointPositionRecorder")
config = load_project_config(project / "profiles/excavator_four_joint_default.json")
adapter = MemoryAdapter((0.0, 0.0, 0.0, 0.0))
limits = {
    "cab": (-178.0, 178.0),
    "boom": (-33.0, 68.0),
    "small_arm": (-78.0, 68.0),
    "bucket": (-88.0, 78.0),
}

controller = MotionController(config, adapter, limits)
print(controller.synchronize())

controller.start_motion(
    {"cab": 1.0, "boom": 1.0, "small_arm": -1.0, "bucket": 0.5},
    {"cab": 2.0, "boom": 4.0, "small_arm": 1.0, "bucket": 1.0},
)

for _ in range(25):
    snapshot = controller.update(0.05)
    print(snapshot.state, snapshot.current_degrees)
```

这个实验把 Profile、Planner、Controller 和 Adapter Protocol 串起来，但仍不依赖 Isaac。

## 9. 基础计算练习

### 练习 1：连续三帧

```text
current = 0°
target = 1°
speed = 2°/s
dt = 0.2 s
```

忽略 Controller 的 `max_update_dt`，直接调用 Planner，三帧输出是什么？

答案思路：每帧最大 0.4°，所以是 0.4°、0.8°、1.0°，最后一帧吸附。

### 练习 2：加入 Controller 截断

同样 `dt = 0.2 s`，但 Controller 的 `max_update_dt = 0.05 s`，speed = 8°/s。第一帧只前进多少？

答案：`8 × min(0.2, 0.05) = 0.4°`。

### 练习 3：安全范围

USD Boom 限位 `[-35°, 70°]`，Profile safety margin 为 2°。目标 -34° 是否可接受？

答案：安全范围是 `[-33°, 68°]`，-34° 会被拒绝。

### 练习 4：预计时长

四个关节距离分别为 `[8°, 20°, 10°, 1°]`，速度 `[8, 5, 5, 1]°/s`。整体多久到达？

答案：各自 `[1, 4, 2, 1]` 秒，整体 4 秒。

### 练习 5：同时到达速度

若希望上题四关节都在 4 秒到达，应设为多少速度？

答案：`[2, 5, 2.5, 0.25]°/s`。若某关节距离为 0，应直接视为 reached；不能把其速度输入 0，因为当前 Planner 要求所有速度为正。

## 10. 代码理解练习

### 练习 A：为什么两次回读

找出 `MotionController.update()` 在 MOVING 分支中的两次 `get_positions_degrees()`，分别写出其用途。

参考思路：第一次得到本帧规划起点；写入后第二次得到运行时接受状态，用于到达判断和记录。

### 练习 B：为什么 Controller 不直接使用 `result.reached`

参考思路：Planner 的 reached 描述命令序列是否走到目标；Controller 要根据 Adapter 回读确认实际运行时状态。

### 练习 C：为什么 Profile Home 在加载时不检查 Stage 限位

参考思路：加载 Profile 时还没有具体 Stage 和它的实际上下限；Home 的有限性可静态检查，范围只能在 Controller 收到 Stage 派生 limits 后检查。

### 练习 D：找出一次记录中止的完整调用链

以“用户切换 Stage”为例：

```text
_on_update 检测 Stage identity 变化
→ _release_binding(abort_recording=True)
→ controller.abort_recording()
→ recorder.abort()
→ partial 被保留
```

## 11. 推荐的进阶功能练习

以下是学习方向，不是本次笔记已经实现的功能。

### 11.1 在 GUI 显示安全限位

目标：每个关节行显示 `[lower, upper]`，并给 FloatDrag 设置范围。

注意点：

- UI 在 Stage 校验前就构建，此时尚不知道实际 limits；
- 绑定不同 Stage 后范围可能变化；
- Controller 校验仍要保留，不能只依赖 UI 限制。

### 11.2 显示预计到达时间

在 `Move all` 前用 `ConstantSpeedPlanner.expected_duration_seconds()` 计算。起点必须使用 Adapter 回读，而不是 GUI 上一次显示值。

### 11.3 增加“同步到达”模式

建议新增明确模式开关：

1. 回读各关节距离。
2. 选择共同持续时间。
3. 计算各自速度。
4. 处理零距离关节。
5. 仍通过 Controller 的正速度和限位校验。

不要改变现有独立速度模式的含义。

### 11.4 记录命令值和原始 `dt`

可增加列：

```text
command_cab, command_boom, ...
raw_dt, used_dt, frame_index
```

先定义清楚：命令值是 Planner 输出，还是 GUI 最终 target；两者不能用同一个含糊字段名。

### 11.5 周期性 durable flush

每 N 帧 `flush + fsync` 可减少断电损失，但会增加磁盘同步开销。应做成可配置策略并测量性能，而不是每帧无条件 fsync。

### 11.6 为 Stage Validator 增加纯测试

可用 `pxr.Usd.Stage.CreateInMemory()` 构造最小 Stage，覆盖：

- 唯一/多个根；
- 缺失关节；
- Drive 冲突；
- 断链和环；
- 质量、惯量和安全范围。

这些测试需要能导入 pxr 的环境，但不一定需要完整 GUI 或 GPU。

### 11.7 完善设计契约检查

按原始设计文档补充：

- PhysicsScene；
- upAxis 和 metersPerUnit；
- 关节 axis；
- 运行时 DOF 类型；
- 运行时与 USD 限位一致性。

新增检查时为每项设计稳定错误码，并为合法/非法 Stage 各写测试。

## 12. 如果要扩展成五关节，哪些地方必须改

当前项目名称和契约都明确是四关节，不能只在 JSON 多加一个对象：

- `ProjectConfig.validate()` 强制 `len(joints) == 4`；
- 默认 Profile、测试和 CSV 契约都是四列关节；
- Stage Validator 的算法本身按 config 循环，较容易泛化，但文档语义是四关节；
- Adapter runtime 要求总 DOF 数等于配置数量；
- GUI 会按 config 动态生成行，布局高度可能需要调整；
- 测试中的 limits、targets、FakeAdapter 初始值全部写了四项；
- 设计文档和 smoke test 的断言/输出都写明 4-DOF。

正确做法是先决定项目要继续保持“四关节挖掘机专用”，还是重构为“N 关节通用 Recorder”，再系统修改契约、命名、测试和文档。
