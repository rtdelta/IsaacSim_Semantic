# 05 语义相机与数据写出

## 1. 从场景标签到训练标签

语义采集路径可以分成六步：

```text
USD Prim 上的 SemanticsLabelsAPI:class
    -> Isaac semantic_segmentation Annotator
    -> runtime ID 图 + idToLabels
    -> 标签标准化
    -> 稳定 uint16 dataset ID 图
    -> 自定义颜色 PNG 与诊断元数据
```

核心原则是：**运行时 ID 是渲染器内部实现细节，标签字符串是桥梁，稳定 dataset ID 才是数据集契约。**

## 2. Stage 中的语义标签

项目统计应用 Schema 名以 `SemanticsLabelsAPI` 开头的 Prim。例如 overlay 可给 Prim 增加：

```usda
over "boom_mesh" (
    prepend apiSchemas = ["SemanticsLabelsAPI:class"]
)
{
    token[] semantics:labels:class = ["boom"]
}
```

默认 `Sim_Fangshan_07_capture_overlay.usda` 只组合原场景，没有额外 author 标签，因此默认运行依赖被组合的 native 场景本身已含正确语义。预检只统计“有语义 Schema 的 Prim 数量”，稳定映射是否覆盖所有真实标签则由 Writer 的严格映射检查负责。

## 3. 语义映射配置

默认 `configs/semantic_mapping_Sim_Fangshan_07_native.json` 定义十类：

| dataset ID | 标签 | RGB 颜色 |
|---:|---|---|
| 0 | `BACKGROUND` | `[0, 0, 0]` |
| 1 | `arm` | `[69, 225, 201]` |
| 2 | `boom` | `[188, 172, 29]` |
| 3 | `bucket_noteeth` | `[190, 68, 110]` |
| 4 | `cab` | `[148, 218, 60]` |
| 5 | `tooth_1` | `[245, 130, 48]` |
| 6 | `tooth_2` | `[145, 30, 180]` |
| 7 | `tooth_3` | `[70, 240, 240]` |
| 8 | `tooth_4` | `[240, 50, 230]` |
| 9 | `tooth_5` | `[210, 245, 60]` |
| 10 | `track` | `[39, 232, 216]` |
| 65535 | `UNLABELLED` | `[255, 0, 255]` |

加载映射时会检查：

- `schema_version == 1`；
- `dataset_dtype == "uint16"`；
- ID、忽略大小写后的标签、颜色三者分别唯一；
- background ID 必须为 0。

代码为全部 65,536 个 `uint16` 值建立颜色 LUT，所以着色时可以直接向量索引，而不必逐像素循环。

## 4. 标签标准化规则

Isaac 的 `idToLabels` 可能是字符串、字典、序列或继承后以逗号连接的标签。`canonical_label()` 递归处理：

- 字典若含当前 `semantic_type`，优先取该键；
- 其他字典/序列取最后一个非空解析结果；
- 字符串按逗号分割，取最后一个非空标签；
- 匹配时忽略大小写；
- 空、background、unlabelled、unlabeled、none 视为背景别名。

例如：

```text
"vehicle, boom"       -> "boom"
{"class": "cab"}     -> "cab"
["machine", "track"] -> "track"
```

“取最后一个标签”用于处理从父级到子级逐渐具体的继承结果：更靠后的标签通常更具体。语义 JSON 中的 `label_resolution` 也明确记录了这一策略。

## 5. Runtime ID 重映射

假设 Annotator 返回：

```text
runtime_ids 中出现 {0, 17, 42}
idToLabels = {
  17: {"class": "boom"},
  42: {"class": "cab"}
}
```

重映射结果：

```text
0  -> BACKGROUND -> dataset ID 0
17 -> boom       -> dataset ID 2
42 -> cab        -> dataset ID 4
```

输出还包含诊断：

```json
{
  "runtime_id_mapping": {
    "17": {
      "source": {"class": "boom"},
      "resolved_label": "boom",
      "dataset_id": 2,
      "dataset_label": "boom"
    }
  },
  "dataset_pixel_counts": {"0": 500000, "2": 120000},
  "unknown_labels": []
}
```

这使每帧都能追查某个临时 ID 最终变成了什么。

## 6. Strict mapping 的真实含义

默认 `--strict-mapping`。遇到映射中没有的标签时，Writer 立即抛出 `KeyError`，生产运行失败。

`--no-strict-mapping` 会允许采集继续，把未知标签写为 65535 和洋红色，并在 `unknown_labels` 中记录。它适合诊断新场景，但当前独立验证器仍会拒绝任何非空 `unknown_labels`。因此：

- 非严格模式是“帮助发现缺失标签”的工具；
- 它不是让带未知标签的数据自动成为合格生产数据的开关。

## 7. 相机调度器为何使用状态机

`SemanticCameraScheduler` 管理一台相机、一个 RenderProduct 和一个 Writer：

1. **CREATED**：只有配置；
2. **RENDER_PRODUCT_READY**：相机已解析，持久 RenderProduct 已创建；
3. **WARMED**：完成渲染预热；
4. **WRITER_ATTACHED**：Writer 开始接收数据；
5. **CAPTURING**：至少完成一帧；
6. **CLOSED**：Writer detach、RenderProduct destroy。

### 7.1 相机解析

若命令行明确给出相机路径，会验证 Prim 类型，并可要求它位于 `cab_root` 下。若传入空路径，则遍历 `cab_root` 子树，要求恰好发现一台相机。

### 7.2 持久 RenderProduct

初始化只创建一次 `SemanticCapture` RenderProduct，并保持 Hydra updates 开启。Warm-up 后也不销毁，以保留渲染器时间历史。

### 7.3 Writer 在预热后挂接

如果预热前就 attach，Replicator 可能把 warm-up 更新当作数据帧。现在的顺序保证第一份文件就是 frame 0000。

## 8. Writer 如何验证输入

`SemanticDatasetWriter` 注册 RGB 和未着色 semantic segmentation Annotator，且只接受一个 RenderProduct。

写一帧时检查：

- 输入存在 `renderProducts`；
- 恰好一个 RenderProduct；
- semantic entry 是字典并包含 ID 图；
- runtime ID 图能转换为二维 `uint32`；
- `idToLabels` 位于 entry 或 `entry.info`；
- RGB 是 `uint8` 的 H×W×3 或 H×W×4；
- RGB 与语义图高宽一致。

语义数据若以 RGBA `uint8` 打包 ID，Writer 会用连续内存视图还原 `uint32`；其他整型形式则显式转换。

## 9. 输出目录详解

一次完整输出结构为：

```text
output/semantic_capture_v3/
├─ run_config.json
├─ semantic_mapping.json
├─ motion_state.jsonl
├─ rgb/
│  ├─ rgb_0000.png
│  └─ ...
├─ semantic_id/
│  ├─ semantic_id_0000.npy
│  └─ ...
├─ semantic_color/
│  ├─ semantic_color_0000.png
│  └─ ...
├─ semantic_runtime_id/             # --save-runtime-ids 时存在
│  ├─ semantic_runtime_id_0000.npy
│  └─ ...
└─ metadata/
   ├─ frame_0000.json
   └─ ...
```

### 9.1 `semantic_id/*.npy`

- 形状：`(height, width)`；
- dtype：`numpy.uint16`；
- 内容：稳定 dataset ID；
- 这是训练与精确统计应使用的主标签文件。

### 9.2 `semantic_color/*.png`

- 形状：H×W×3；
- dtype：`uint8`；
- 内容：通过映射 LUT 从 semantic ID 精确生成；
- 用于可视化，不应反向做颜色近似分类。

### 9.3 `semantic_runtime_id/*.npy`

- dtype 统一为 `uint32`；
- 只用于调试 Isaac 当次运行分配的 ID；
- 可通过 `--no-save-runtime-ids` 关闭，验证器会据清单调整所需文件数。

### 9.4 `rgb/*.png`

RGB/RGBA 标注原样作为 `uint8` 图像保存。验证器会检查分辨率，但不会把 RGB 与语义边界做视觉配准推断；同步性依赖共享 RenderProduct 和上下文协议。

### 9.5 `metadata/frame_XXXX.json`

每帧包含：

- schema version 2；
- frame ID、dataset/timeline time、physics step；
- 相机路径与 16 元素世界矩阵；
- 运动命令、实际位置、误差、刚体矩阵；
- RenderProduct 名和分辨率；
- runtime ID 映射、像素计数、未知标签。

### 9.6 `motion_state.jsonl`

一行一个 JSON 对象，适合流式追加与逐行处理。它比 per-frame metadata 多出：

- 总 `simulation_time`；
- `WorldScheduler.get_state()`；
- 完整相机状态和光学参数；
- `CaptureReceipt`，即本帧计划写出的相对路径。

每写一行立即 `flush()`，所以失败运行通常仍能保留已完成帧的状态。

### 9.7 `run_config.json`

这是运行级权威清单，重点字段：

```text
status                 running / complete / failed
inputs                 输入文件元信息与 SHA-256
render                 profile、requested/effective、sampling、mismatches
preflight              Stage 与 Articulation 检查
motion_control         profile、bootstrap/setup、binding
trajectory / joints    轨迹摘要与安全范围
writer                 pending/completed
error                  失败时的异常类型与消息
```

### 9.8 `semantic_mapping.json`

Writer 初始化时把实际使用的映射副本写进输出目录。训练代码应优先绑定这份随数据集归档的映射，而不是假设项目配置永远不变。

## 10. 文件名与回执

帧名固定为四位零填充：`0000`、`0001`。每帧保存后构造 `CaptureReceipt`：

```json
{
  "frame_id": 0,
  "rgb": "rgb/rgb_0000.png",
  "semantic_id": "semantic_id/semantic_id_0000.npy",
  "semantic_color": "semantic_color/semantic_color_0000.png",
  "semantic_runtime_id": "semantic_runtime_id/semantic_runtime_id_0000.npy",
  "metadata": "metadata/frame_0000.json"
}
```

Receipt 被写入 `motion_state.jsonl`，让后续消费者不必自己拼接路径。

## 11. 一个简单读取示例

在普通 Python 中读取某帧：

```python
import json
from pathlib import Path

import numpy as np
from PIL import Image

root = Path("output/semantic_capture_v3")
frame = "0000"

ids = np.load(root / "semantic_id" / f"semantic_id_{frame}.npy")
color = np.asarray(Image.open(root / "semantic_color" / f"semantic_color_{frame}.png"))
rgb = np.asarray(Image.open(root / "rgb" / f"rgb_{frame}.png"))
metadata = json.loads((root / "metadata" / f"frame_{frame}.json").read_text("utf-8"))

print(ids.shape, ids.dtype)
print(np.unique(ids, return_counts=True))
print(metadata["dataset_time"], metadata["motion"]["actual_degrees"])
```

这个示例适合探索；正式使用前仍应先运行项目提供的完整验证器。
