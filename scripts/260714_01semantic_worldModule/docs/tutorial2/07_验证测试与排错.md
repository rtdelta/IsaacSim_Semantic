# 07 验证、测试与排错

## 1. 四层防线

项目不是等采完后才发现问题，而是逐层阻断：

| 层 | 发生时机 | 代表模块 | 检查内容 |
|---|---|---|---|
| 配置验证 | 启动 Isaac 前 | `CaptureTiming`、profiles | JSON schema、参数、路径、频率 |
| Stage 预检 | Stage 组合后 | 两个 validator | 相机、语义、资产、Articulation |
| 运行时断言 | 每个物理步/采集帧 | scheduler、motion、ledger | 时间、冻结、回读、帧归属 |
| 输出验收 | 运行完成后 | `validate_semantic_output.py` | 文件、像素、清单、状态、移动 |

理解错误属于哪一层，比记住每个异常字符串更重要。

## 2. `run_config.json` 排错顺序

遇到失败时按以下顺序读：

1. `status`：是 `failed`、`running` 还是 `complete`；
2. `error.type` 与 `error.message`：首次阻断原因；
3. `preflight.issues`：通用 Stage 错误；
4. `preflight.articulation.issues`：运动结构错误；
5. `render.mismatches` 和 `render.effective`：渲染设置是否生效；
6. `motion_control.bootstrap_steps` 与 `binding`：Tensor 是否 ready、DOF 是否匹配；
7. `writer.pending/completed`：Writer 是否完整消费帧。

若状态仍是 `running`，通常意味着进程被强制终止，来不及进入异常处理。此时结合最后一行 `motion_state.jsonl` 和控制台日志判断停止在哪帧。

## 3. 输出验证器逐项做什么

运行：

```bash
/root/isaacsim/python.sh validate_semantic_output.py \
  --output <output-dir> \
  --mapping <mapping.json> \
  --expected-frames <N>
```

### 3.1 Manifest v2 基础契约

- `status == complete`；
- 清单帧数等于期望；
- 渲染器 requested/effective/profile 三者一致；
- 采样模型与渲染器匹配；
- strict Stage 运行必须通过通用预检；
- Writer pending 为 0，completed 等于帧数；
- 重新建立 `CaptureTiming`。

### 3.2 Manifest v3 运动契约

- 控制模式是 `articulation_direct_position`；
- 逻辑关节顺序精确匹配四列；
- 回读容差有效；
- motion 时 Articulation 预检通过；
- bootstrap steps 是非负整数；
- setup steps 恰好为 1；
- adapter 在清单中是 bound 且 ready；
- DOF 顺序与逻辑关节顺序一致。

### 3.3 文件与像素契约

逐类目录的文件数必须恰好等于 N。逐帧检查：

- semantic ID 形状为 `(height, width)`、dtype 为 `uint16`；
- 所有 ID 都在 background/classes/unknown 定义集合中；
- 颜色 PNG 必须逐像素精确等于映射 LUT 结果；
- RGB 分辨率正确；
- metadata frame ID、分辨率、时间正确；
- `unknown_labels` 必须为空。

### 3.4 运动与同步契约

- `motion_state.jsonl` 行数等于 N；
- 行内 frame ID 顺序正确；
- metadata 与 state 的 timeline time、physics step 一致；
- commanded、actual、error 各有四个有限值；
- `error == actual - commanded`；
- 误差不超过 profile 容差；
- commanded 与 actual 都在安全限位内；
- motion 模式至少一个刚体矩阵变化，相机矩阵也变化；
- static 模式刚体与相机都不变化。

验证器不只是检查“文件能打开”，而是在验证这个数据集对自身描述是否自洽。

## 4. 常见错误与定位

| 症状/错误 | 常见原因 | 首先检查 | 处理方向 |
|---|---|---|---|
| `file not found` | 相对路径解析错误或文件没同步 | 清单 input path | 路径相对项目根，或传绝对路径 |
| `ASSET_UNRESOLVED` | subLayer/USD 依赖不存在 | overlay 中绝对路径 | 在远端补齐资产或改 overlay |
| `RENDER_ASSET_UNRESOLVED` | 纹理/HDR 缺失 | preflight warning | 可诊断运行，生产前评估画质影响 |
| `CAMERA_INVALID` | Prim 不存在或不是 Camera | `--camera` | 用正确 USD 路径 |
| `CAMERA_PARENT_INVALID` | 相机不在 cab 下 | camera/cab-root | 运动数据用车载相机；诊断时显式放宽 |
| `SEMANTICS_MISSING` | 场景没有标签 Schema | Stage/overlay | 给目标 Prim author 语义标签 |
| `INVALID_ROOT_HINT` | 根路径没有 Articulation API | joint profile | 改为真实根或修资产 |
| `AMBIGUOUS_JOINT` | 候选名匹配多个 Prim | articulation issues | 增加唯一 candidate path |
| `DRIVE_CONFLICT` | 关节仍有 Angular Drive | USD applied schemas | 移除 Drive 或改控制架构，不能双控 |
| `BROKEN_CHAIN` | body0/body1 顺序错误 | joint relations | 修 USD 关节拓扑 |
| `INVALID_MASS/INERTIA` | 刚体物理参数缺失或非正 | body Prim 属性 | 在资产中 author 合法值 |
| tensor not ready timeout | Timeline/物理初始化失败 | bootstrap_steps、日志 | 检查根、刚体、PhysX，必要时提高有界超时 |
| `Missing DOFs` | profile 名与 wrapper 不同 | available_dof_names | 修候选/实际 DOF 名 |
| readback exceeded tolerance | 直接命令未被接受或有竞争控制 | commanded/actual/error | 查 Drive、DOF 顺序、限位和物理配置 |
| unknown semantic labels | mapping 漏类或标签拼写不同 | frame metadata mapping | 把真实标签加入新映射版本 |
| render setting mismatch | Carb 设置被拒绝/覆盖 | render requested/effective | 核对 Isaac 版本支持和键名 |
| output directory not empty | 复用了目录 | `--output` | 使用新目录；谨慎决定是否 overwrite |
| contains more frame files | overwrite 未清理旧帧 | 目录文件数 | 换新目录或准确清理旧目标 |
| schedule moved backwards | 时间策略被修改破坏单调性 | timing/loop | 恢复单调 frame -> step 映射 |
| camera did not move | 相机不是 cab 子级或父变换未生效 | camera matrix | 修相机层级/Stage 变换 |

## 5. `strict` 开关的边界

### 5.1 `--no-strict-stage`

它允许通用预检中的非致命 error 不在 `raise_if_blocking()` 阶段阻断，但以下情况仍不能绕过：

- 源 Stage 缺失；
- mapping 缺失；
- 相机无效；
- 完全没有语义标签；
- motion 启用时 Articulation 报告有 error；
- 后续相机调度器自身的结构要求。

所以它是诊断开关，不是“忽略所有错误”。

### 5.2 `--no-strict-mapping`

Writer 会保存未知类为 65535，但正式验证仍拒绝非空 `unknown_labels`。正确流程是：诊断一次，读取实际标签，更新新的 mapping 配置，再重新生产。

### 5.3 `--overwrite`

它不清空输出，只允许 DiskBackend 写入。短帧重跑会残留长帧旧文件。生产中建议不用它，改用不可复用的新目录名。

## 7. 单元测试没有覆盖什么

即使 69 个测试全部通过，也不能证明：

- 远端 subLayer 和纹理真实存在；
- 当前 Isaac Sim 版本接受所有 Carb setting；
- GPU/驱动能运行所选 renderer；
- Replicator 回调在真实环境中与预期数据形状完全一致；
- 真实 Articulation tensor 能在超时内 ready；
- 真实输出的视觉内容符合业务目标。

因此最低集成验证是“3 帧 smoke capture + 完整输出 validator”。

## 8. 如何为修改补测试

### 8.1 修改纯数学/配置

直接在对应测试中增加最小正例和反例。例如增加新时间策略时至少验证：

- 第 0、1、末帧；
- 非法输入；
- 静态模式；
- 与 `validate_semantic_output` 的重算一致。

### 8.2 修改 Isaac 边界

优先注入 fake 对象，不要让单元测试依赖启动 Isaac。例如 `IsaacArticulationAdapter` 支持传入 `articulation_factory`，就是为了测试真实边界协议。

### 8.3 修改输出 schema

必须同时更新：

1. Writer/manifest 生成；
2. `validate_semantic_output.py`；
3. 最小数据集测试；
4. 本教程中的输出格式说明；
5. schema version 与兼容分支。

只修改写出端、不修改验证端，会让新数据无法验收；直接放宽验证则可能让旧错误混入。

## 9. 画质问题的排查顺序

图像过黑、模糊或有噪点时不要先盲目加 subframes：

1. 确认两次运行 Stage、相机矩阵、分辨率和帧时刻一致；
2. 查看 `render.renderer.effective` 是否正确；
3. 查看 `render.mismatches`；
4. 查看相机 optics 与曝光属性；
5. 查看 preflight 中缺失纹理/环境图 warning；
6. 确认 motion blur、depth of field 的实际策略；
7. 再用 `compare_render_quality.py` 对全图或 ROI 比较；
8. 最后调整 profile，而不是临时在入口硬编码 setting。
