# 08 源码精读与二次开发

## 1. 最有效的源码阅读顺序

不要从入口第一行一路读到底。建议按“纯逻辑 -> 状态机 -> 边界 -> 编排 -> 验收”阅读：

1. [`capture_timing.py`](../../capture_timing.py)：20 多行纯数学；

4. [`joint_control_profile.py`](../../joint_control_profile.py)：配置契约；
5. [`render_profile.py`](../../render_profile.py)：另一种配置契约与 read-back；
6. [`world_scheduler.py`](../../world_scheduler.py)：世界状态机；
7. [`excavator_joint_motion.py`](../../excavator_joint_motion.py)：轨迹与每步钩子；
8. [`semantic_dataset_writer.py`](../../semantic_dataset_writer.py)：帧如何变成文件；

“先读验证器”也是很好的第二遍方法，因为它告诉你哪些输出字段是真正受保护的，哪些只是诊断信息。

## 4. 代码中的设计模式

### 4.1 不可变值对象

`CaptureTiming`、`FrozenWorldSnapshot`、`CaptureContext`、`CaptureReceipt`、profiles 都使用 frozen dataclass。好处是配置与帧状态在跨模块传递时不会被修改。

### 4.2 状态机

World 与 Camera 都通过枚举约束生命周期。状态检查把潜在的隐式错序转换成显式异常。

### 4.3 Adapter

`IsaacArticulationAdapter` 隔离 Isaac 张量返回类型、角度单位和运行时导入。业务层只处理有顺序的 degree tuple。

### 4.4 Profile + Read-back

配置不只“写入”，还要“读回”。`RenderProfileManager` 把运行时实际设置作为一等数据，避免把希望使用的参数误认为已生效参数。

### 4.5 Ledger/Receipt

这是一个小型事务账本：请求先 arm，回调 consume，完成后生成 receipt。它比用 Writer 内部 frame counter 更能保护业务帧身份。

### 4.6 Validator as executable specification

输出验证器就是数据集 schema 的可执行说明。写出端与验证端分离，可以发现文件写全但语义不自洽的问题。

## 5. 扩展一：增加或修改语义类别

最安全流程：

1. 在新的 overlay 或原 Stage 中给目标 Prim 添加 `SemanticsLabelsAPI:class`；
2. 复制业务配置，减少 `frames` 并暂时设置 `"strict_mapping": false`；
3. 从 `metadata/frame_XXXX.json -> unknown_labels` 和 `runtime_id_mapping` 获取真实标签字符串；
4. 复制现有 mapping 为新版本；
5. 给新类分配未使用的 `uint16` ID 和唯一 RGB；
6. 更新 `class_count`、`semantic_prim_count` 等描述信息；
7. 恢复 strict mapping 重新采集；
8. 执行完整验证器。

不要改变已有 ID 的含义。若 ID 2 过去代表 boom，新版本也应保持，除非你明确创建一个不兼容的新数据集 schema 并隔离训练流程。

## 6. 扩展二：更换 Stage 或相机

1. 创建新的 overlay，让大资产保持只读；
2. 避免把仅在某一台机器存在的绝对路径直接当成可移植配置；
3. 为新 Stage 建立匹配的 mapping JSON；
4. 检查相机 Prim 类型、层级和 optics；
5. 若 motion，更新 joint profile 的根、候选名/路径；
6. 先运行 static 诊断，再运行 motion；
7. 比较 `preflight`、相机矩阵和 Articulation binding。

当前 `source_usd` 字段只在 mapping 中作为元数据记录，实际 Stage/mapping 是否属于同一资产主要依赖操作者选择和标签严格匹配。若项目需要更强保证，可以在入口增加 source identity 契约并在清单中校验。

## 7. 扩展三：增加新的渲染 profile

优先复制一个 schema v2 profile，再修改：

- `name`；
- `renderer`；
- `launch_settings`；
- `capture_settings`；
- `settings`；
- `required_settings`；
- 描述性 `metadata`。

RealTimePathTracing 规则：

- 必须配置且只配置一个 DLSS execMode 键；
- `anti_aliasing=3`；
- 不得定义 PathTracing SPP。

PathTracing 规则：

- 不得定义 DLSS execMode；
- 必须有 `spp` 与 `totalSpp`，并加入 required；
- `spp` 为 1～32；
- launch 的 samples per pixel 必须等于 setting 的 spp；
- 必须明确 denoiser；
- `anti_aliasing=0`；
- 必须要求 animation time change 时重置积累。

若要增加新的 SimulationApp launch key，还需更新 `render_profile.py` 的 `ALLOWED_LAUNCH_SETTINGS` 并补测试。若只是新的 Carb setting，放入 `settings`；真正决定正确性的项加入 `required_settings`。

## 8. 扩展四：增加世界属性调度

`WorldScheduler.update(simulation_time)` 是当前预留扩展点，可用于在**下一个物理步之前**更新灯光、材质或环境。

扩展时要做三件事：

1. 输入只依赖明确的 dataset time，避免 wall-clock 随机性；
2. 更新发生在 before-step 之前或明确规定与关节命令的顺序；
3. 把影响训练的世界状态加入 `CaptureContext`/manifest，并扩展验证器。

只改变场景而不记录，会让数据不可复现。若使用随机化，还应记录固定 seed 和每帧采样结果，而不能只记录分布参数。

## 9. 扩展五：从四关节改为其他机械结构

这不是只改 JSON 就能完成。当前“四关节”在多处是显式契约：

- `joint_control_profile.py` 的 `EXPECTED_JOINT_COUNT=4`；
- `excavator_joint_motion.py` 的 `JOINT_NAMES`；
- CSV 固定表头；
- Articulation validator 要求四关节、五刚体链；
- validation schema 要求固定逻辑名；
- 测试和文档都绑定此顺序。

通用化建议：

1. 让 profile 成为逻辑名顺序的唯一来源；
2. CSV parser 按 profile 动态生成列；
3. validator 按 N 个关节检查 N+1 链；
4. output validator 从清单读取 N，而非硬编码；
5. 提升 manifest schema version；
6. 同时保留 v3 四关节兼容分支；
7. 用另一种 2～3 DOF fake 结构补通用化测试。

## 10. 扩展六：异步或多相机采集

当前实现故意一次阻塞一帧、只允许一个 RenderProduct。这保证：

```text
FIFO 头部上下文 == 下一次 Writer 回调 == 当前 frame ID
```

改成异步/多相机后可能出现回调乱序，必须先重新设计身份协议：

- 每个相机独立 Ledger，或上下文包含 camera/render-product ID；
- 回调携带可验证的请求 token；
- 一个 frame 的多相机输出必须有聚合 receipt；
- 完成条件从单回执变成 barrier；
- 世界必须直到所有相机完成后才能恢复；
- 文件命名和 validator 必须加入 camera 维度。

不要只删除 `wait_until_complete()` 来“提高速度”。那会先破坏正确性，再产生难以复现的串帧。

## 11. 当前实现的边界与注意点

### 11.1 远端路径可移植性

默认 overlay 指向 `/root/gpufree-data/...`。项目目录复制到别处并不自动带走大资产。更可移植的做法是使用统一挂载点、资产解析器或部署时生成 overlay。

### 11.2 `--overwrite` 不删除旧文件

入口只允许继续写，DiskBackend 也只覆盖同名文件。短任务覆盖长任务会留下多余帧。

### 11.3 JSONL 不是原子整体

Manifest 每次原子替换，`motion_state.jsonl` 则逐行追加和 flush。失败时它可能是合法的部分数据，这是诊断优势；但只有 status complete 且 validator PASS 的完整目录才能交付。

### 11.4 非严格 mapping 不等于合格数据

未知标签可以保存为 65535，但 validator 仍拒绝。该模式仅用于发现标签。

### 11.5 单元测试不启动真实 Isaac

Fake 对象保护 API 协议和逻辑，但 GPU、真实 Stage composition、真实 Replicator shape 仍需要 smoke integration。

### 11.6 一个应用更新等于一个计数物理步的假设

`WorldScheduler.step()` 在固定时间步设置下调用一次 `app.update()` 后手工加一。若未来 Kit 调度语义变化或引入不同 stepping API，应增加真实运行时核对，不要仅相信内部计数。

### 11.7 画质工具不证明可比性

`compare_render_quality.py` 的指标只衡量传入像素。它不会验证两个文件对应同一个 Stage、相机和时间。

## 12. 推荐的改进路线

从风险和收益看，可按顺序推进：

1. 增加一个真实 Isaac 的 1 帧集成测试脚本；
2. 给每次输出自动生成唯一 run ID，降低目录复用风险；
3. 增加 mapping 与 Stage identity 的显式匹配；
4. 把关键输出 schema 写成 JSON Schema；
5. 增加每帧文件哈希或运行结束后的数据集索引；
6. 扩展世界随机化，同时记录 seed 与采样值；
7. 在保持单帧 barrier 的前提下探索多相机；
8. 最后才考虑跨帧异步流水线。

每一步都应以“validator 能否证明新不变量”为完成标准，而不只是“代码跑起来了”。
