# 01 零基础概念：从 USD 场景到语义数据集

## 1. Isaac Sim 在本项目中的角色

Isaac Sim 可以粗略理解为四套能力的组合：

- **USD 场景系统**：描述场景层级、网格、相机、刚体、关节和语义标签；
- **PhysX 物理系统**：让刚体与关节在离散时间步中更新；
- **RTX 渲染系统**：根据相机生成 RGB 图像；
- **Replicator 数据系统**：从 RenderProduct 获取 RGB、语义分割等标注，并交给 Writer 保存。

本项目没有使用 Isaac Sim GUI 去手动播放和截图，而是由 Python 精确控制这些系统的调用顺序。

## 2. Kit、`SimulationApp` 与普通 Python

Isaac Sim 建立在 Omniverse Kit 上。`SimulationApp` 是启动 Kit 运行时的入口。只有它启动以后，很多模块才真正可用，例如：

```python
from isaacsim import SimulationApp
simulation_app = SimulationApp(launch_config=...)

import omni.usd
import omni.replicator.core as rep
import carb.settings
from pxr import UsdGeom
```

因此入口脚本采用了非常重要的加载顺序：

1. 先用普通 Python 解析参数、JSON 配置并验证文件；
2. 再创建 `SimulationApp`；
3. 最后导入依赖 Kit/Replicator 的模块。

这也是为什么一些模块能在普通 Python 中测试，而 `semantic_capture_custom.py` 不能直接在没有 Isaac Sim 的解释器中运行。

## 3. USD 的最小知识集

### 3.1 Stage

Stage 是组合完成后的整个场景。它像一棵带属性的树，里面包含相机、网格、刚体和关节。

本项目通过：

```python
omni.usd.get_context().open_stage(args.usd)
stage = omni.usd.get_context().get_stage()
```

打开 Stage。打开调用返回后，资源仍可能正在异步组合，所以 `wait_for_opened_stage()` 会持续 `simulation_app.update()`，直到：

- Stage 对象存在；
- 根层路径与请求文件匹配；
- 待加载数量为 0；
- 伪根节点下已经有子 Prim。

### 3.2 Prim 与路径

Prim 是 USD 场景树中的节点，使用绝对路径标识。例如：

```text
/root/Xform/operator_cab_mesh/Camera_01
/World/Joints/track_operator_cab_joint
```

同一个 Prim 可以有类型、属性、关系和应用的 API Schema。项目会检查：

- 相机 Prim 是否是 `UsdGeom.Camera`；
- 关节 Prim 是否是 `PhysicsRevoluteJoint`；
- 刚体是否应用 `RigidBodyAPI`、`MassAPI`；
- 语义 Prim 是否应用 `SemanticsLabelsAPI:class`。

### 3.3 Layer 与组合

USD Stage 可以由多个 Layer 组合。默认文件 `configs/Sim_Fangshan_07_capture_overlay.usda` 本身很小，只声明了一个 subLayer：

```usda
#usda 1.0
(
    subLayers = [
        @/root/gpufree-data/wyb/StageMaterial02/Sim_Fangshan_07.usda@
    ]
)
```

这表示真正的大场景仍位于远端绝对路径。当前文件是一个“叠加入口”。如果远端机器没有该路径，Stage 组合会出现未解析依赖，项目不能完成生产采集。

其他 overlay 文件还使用 `over` 给已有 Prim 增加语义标签。`over` 不重新定义完整对象，只在较强 Layer 中补充或覆盖意见。这种方式适合让原始资产保持不变，把“数据集专用标签”放在单独文件中。

### 3.4 Attribute、Relationship 与 API Schema

- **Attribute** 是值，例如 `physics:mass`、`physics:lowerLimit`。
- **Relationship** 指向其他 Prim，例如关节的 `physics:body0` 与 `physics:body1`。
- **API Schema** 给 Prim 增加一组行为和属性，例如 `PhysicsArticulationRootAPI`、`SemanticsLabelsAPI:class`。

四关节链能否正确控制，主要取决于关系是否形成：

```text
固定根 -> track -> cab -> boom -> small_arm -> bucket
```

每个 RevoluteJoint 的 `body0` 应是上一节，`body1` 应是下一节。

## 4. 物理世界的核心概念

### 4.1 Rigid Body、Joint、DOF

- **Rigid Body（刚体）**：具有质量、惯量和位姿的物体；
- **Joint（关节）**：约束两个刚体之间的相对运动；
- **DOF（自由度）**：关节允许独立变化的运动量。

本项目的四个 RevoluteJoint 各有一个旋转 DOF：

| 逻辑名 | 默认候选 DOF/Prim 名 |
|---|---|
| `cab` | `track_operator_cab_joint` |
| `boom` | `platform_boom_joint` |
| `small_arm` | `boom_small_arm_joint` |
| `bucket` | `small_arm_bucket_joint` |

### 4.2 Articulation

Articulation 是由关节连接的刚体系统。相比把每个关节当作互不相关的对象，它能以统一顺序读取和设置全部 DOF。

本项目使用 `isaacsim.core.experimental.prims.Articulation`，一次批量提交四个位置。配置要求固定底座，并禁止关节上的 Angular Drive。原因是项目选择了“直接设置 Articulation 位置”的唯一控制源；如果 Angular Drive 同时工作，就会有两个控制器争夺同一 DOF。

### 4.3 角度单位

CSV、配置和输出元数据全部使用 **degree（度）**，Isaac 张量接口使用 **radian（弧度）**。适配器在边界处完成转换：

\[
\text{radian}=\text{degree}\times\frac{\pi}{180}
\]

统一在业务层使用度，可以减少手写轨迹和读日志时的认知负担。

## 5. Timeline、物理步与渲染帧

这三个概念不能混为一谈：

- **Timeline**：Kit 的播放时间轴，可以 play、pause；
- **物理步**：PhysX 更新一次世界状态，本项目固定为 1/60 秒；
- **渲染更新**：生成或积累一次图像，不一定让物理世界前进。

项目在采集时暂停 Timeline，再调用 Replicator：

```python
rep.orchestrator.step(
    rt_subframes=...,
    delta_time=0.0,
    pause_timeline=False,
)
```

`delta_time=0.0` 的目的就是“在同一世界状态上渲染，不推进数据集时间”。`rt_subframes` 是一个输出帧内部的渲染采样/时间子帧数量，不是物理步数量。

## 6. 相机与 RenderProduct

USD Camera Prim 保存相机的位姿和光学参数。RenderProduct 可以理解为“某台相机以某个分辨率产生的渲染输出”。本项目整个运行只创建一个持久 RenderProduct：

```python
rep.create.render_product(
    camera_path,
    resolution=(width, height),
    name="SemanticCapture",
)
```

“持久”很重要，因为重复创建会破坏渲染器的时间历史、增加开销，也更难保证 Writer 与帧一一对应。

项目还记录相机的 4×4 世界变换矩阵。矩阵被展平为 16 个浮点数写入 JSON，用于复现视角、
判断 Camera 是否移动，以及校验图像与帧状态的一致性。

## 7. Replicator 的三个对象

### 7.1 Annotator

Annotator 从 RenderProduct 读取某种标注。本项目使用：

- `rgb`：RGB 或 RGBA `uint8` 图像；
- `semantic_segmentation`：未着色的运行时语义 ID，以及 `idToLabels`。

### 7.2 Writer

Writer 接收 Annotator 输出并决定如何保存。项目自定义 `SemanticDatasetWriter`，而不是直接使用内置 Writer，因为它需要：

- 将运行时 ID 映射成稳定 `uint16` ID；
- 生成自定义颜色图；
- 写入每帧权威上下文；
- 使用帧号而非 Writer 内部计数命名；
- 返回可核验的文件回执。

### 7.3 Backend

`DiskBackend` 负责实际写文件。Writer 通过 `backend.schedule(...)` 调度 NPY、PNG、JSON 写入，采集器随后调用 `wait_until_complete()` 等待完成。

## 8. 语义分割：标签、运行时 ID、数据集 ID

假设场景中某个 Prim 标注为 `boom`。Isaac 在某次运行里可能把它分配为运行时 ID 17，另一次运行可能是 9。如果直接保存 17，训练集标签就不稳定。

本项目建立两步映射：

```text
runtime ID 17 -> idToLabels 中的 "boom" -> dataset ID 2
```

在默认 `Sim_Fangshan_07_native` 映射中：

- 背景固定为 0；
- 正式类别固定为 1～10；
- 未知类别固定为 65535，并使用洋红色 `[255, 0, 255]`；
- 输出数组类型固定为 `uint16`。

颜色 PNG 只是方便人查看，训练和精确判断应以 `semantic_id/*.npy` 为准。

## 9. 配置契约与运行清单

项目中的 JSON 不只是参数集合，而是带版本和校验规则的契约：

- 语义映射要求 `schema_version=1`、`dataset_dtype=uint16`，ID/标签/颜色不能重复；
- 业务启动配置要求 `schema_version=1`，字段完整、类型准确、相对路径以配置文件目录为基准；
- 关节配置要求恰好四个逻辑关节，候选名和候选路径不能冲突；
- 渲染配置区分 schema v1/v2，并检查渲染器特有设置；
- 运行结果使用 `run_config.json` schema v4 记录业务配置哈希、有效配置、输入哈希、实际渲染设置、预检结果和完成状态。

“配置文件 + SHA-256 + 实际生效值”让一次数据生产可以追溯。只保存命令行是不够的，因为同一路径下的文件内容可能后来发生变化。
