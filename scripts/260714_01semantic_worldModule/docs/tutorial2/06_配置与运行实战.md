# 06 配置与运行实战

## 1. 先判断你在哪种环境

项目实际有两种运行环境：

| 环境 | 能做什么 | 不能做什么 |
|---|---|---|
| 普通 Python（当前 Windows 工作区等） | 跑单元测试、加载配置、验证已有输出、比较图像 | 不能启动没有安装的 Isaac/Kit/Replicator |
| 安装了 Isaac Sim 的 Linux/远端环境 | 完整打开 USD、物理仿真、RTX 渲染和数据采集 | 仍需保证绝对 USD 资产路径真实存在 |

项目没有独立的 `requirements.txt`。普通 Python 测试/验收主要用到：

```text
Python 3
numpy
Pillow
pytest
```

完整运行依赖 Isaac Sim 自带的 `isaacsim`、`omni`、`carb`、`pxr` 和 Replicator 模块，应使用 Isaac Sim 的 Python 启动脚本，不建议在系统 Python 中单独拼装这些包。

PowerShell：

```powershell

```

当前源码的验证结果为：


```powershell

```

`simulation_orchestrator.py --help` 能在普通 Python 中显示，是因为帮助退出发生在 `SimulationApp` 导入前；真正采集仍需要 Isaac Sim。

## 3. 远端启动包装做了什么

`run_capture_remote.sh` 会在项目下建立 `.runtime`，并把 HOME、TMP、XDG cache/config、CUDA cache 和 OptiX cache 指到这里，然后执行：

```bash
/root/isaacsim/python.sh simulation_orchestrator.py "$@"
```

这提供相对隔离、可写的运行目录，也让无交互远端会话不依赖用户真实 HOME。

注意：脚本中的 Isaac Sim 路径固定为 `/root/isaacsim/python.sh`。如果远端安装位置不同，应先修改启动包装或直接使用实际 Python 路径。

## 4. 运行前必须确认的远端资产

默认 overlay 包含绝对 subLayer：

```text
/root/gpufree-data/wyb/StageMaterial02/Sim_Fangshan_07.usda
```

因此至少确认：

```bash
test -f /root/gpufree-data/wyb/StageMaterial02/Sim_Fangshan_07.usda
test -x /root/isaacsim/python.sh
```

还要确认组合后的 Stage 中存在：

```text
相机 /root/Xform/operator_cab_mesh/Camera_01
根   /World/Joints/world_track_fixed_joint
四个配置中的关节
至少一个 SemanticsLabelsAPI 标签
```

这些内容不需要人工全部猜测；Stage preflight 和 Articulation validator 会给出结构化错误。但先检查绝对路径能节省大量加载时间。

## 5. 第一次 smoke test

在远端项目目录中，建议使用一个**全新**输出目录：

```bash
./run_capture_remote.sh \
  --frames 3 \
  --width 640 \
  --height 360 \
  --output output/tutorial_smoke_001
```

这仍使用默认运动、默认相机、默认 RealTimePathTracing profile，只降低帧数和分辨率。

成功后应看到：

```text
[simulation-orchestrator] Captured frame 3/3 ...
[semantic-capture] Complete: .../output/tutorial_smoke_001
```

然后验证：

```bash
/root/isaacsim/python.sh validate_semantic_output.py \
  --output output/tutorial_smoke_001 \
  --mapping configs/semantic_mapping_Sim_Fangshan_07_native.json \
  --expected-frames 3
```

最终期望：

```text
[validation] PASS: 3 frames, schema=3, ...
```

## 6. 不同目的的命令模板

### 6.1 默认运动采集

```bash
./run_capture_remote.sh \
  --frames 50 \
  --output output/motion_realtime_run_001
```

默认时间为 t=0.0～4.9 秒。轨迹长度为 5 秒，默认 `hold`，但这 50 帧尚未超过轨迹末端。

### 6.2 循环轨迹

默认轨迹首尾闭合，可使用：

```bash
./run_capture_remote.sh \
  --frames 101 \
  --trajectory-mode loop \
  --output output/motion_loop_run_001
```

101 帧、10 FPS、包含初始帧，对应 t=0～10 秒，完整覆盖两个 5 秒周期边界。

### 6.3 静态重复采集

```bash
./run_capture_remote.sh \
  --capture-mode static \
  --frames 8 \
  --output output/static_run_001
```

所有帧 `dataset_time=0`，物理和相机都不应变化。适合比较渲染噪声或检查标签覆盖。

### 6.4 使用驾驶室外的诊断相机

```bash
./run_capture_remote.sh \
  --capture-mode static \
  --camera /root/Camera_03 \
  --no-require-camera-below-cab \
  --frames 3 \
  --output output/external_camera_diagnostic_001
```

只有明确做静态诊断时才建议关闭驾驶室父级约束。运动数据若使用静态外部相机，当前验证器会因“camera did not move with the cab”拒绝。

### 6.5 自动发现驾驶室下唯一相机

Bash 中传空字符串：

```bash
./run_capture_remote.sh \
  --camera "" \
  --frames 3 \
  --output output/discovered_camera_001
```

如果 cab root 下不是恰好一台相机，会明确失败。

### 6.6 高质量 Path Tracing

```bash
./run_capture_remote.sh \
  --renderer PathTracing \
  --render-profile configs/render_pathtracing_720p_64spp.json \
  --frames 3 \
  --output output/pathtracing_run_001
```

该 profile 每个渲染帧 8 spp，8 个 RT subframes，计划每输出帧 64 spp，启用 denoiser。计算成本显著高于默认实时模式，先做 1～3 帧试跑。

### 6.7 覆盖 profile 的采集参数

```bash
./run_capture_remote.sh \
  --rt-subframes 4 \
  --warmup-render-frames 4 \
  --frames 2 \
  --output output/fast_diagnostic_001
```

这是诊断配置，不代表生产画质。覆盖值会写入清单，但 profile 源文件 SHA 不变；`capture_settings` 记录实际使用的覆盖结果。

### 6.8 不保存临时运行时 ID

```bash
./run_capture_remote.sh \
  --no-save-runtime-ids \
  --frames 3 \
  --output output/no_runtime_ids_001
```

稳定 `semantic_id` 和颜色图仍会保存，只有 `semantic_runtime_id` 目录不再是验收必需项。

## 7. 渲染 profile 怎么选

| Profile | 渲染器 | 核心采样 | 适合 |
|---|---|---|---|
| `render_realtime_pathtracing_720p.json` | RealTimePathTracing | DLSS Quality，16 RT subframes | 默认运动数据生产候选 |
| `render_pathtracing_720p_64spp.json` | PathTracing | 8 spp × 8 subframes = 64 nominal spp | 少量高质量/静态图 |
| `render_quality_dlss_720p.json` | 旧 RaytracedLighting | schema v1 legacy | 兼容历史实验，不是当前默认正式路径 |

显式 `--renderer` 必须与 profile 内的 `renderer` 一致。若只传 `--renderer PathTracing` 而不传 profile，入口会自动选择对应默认 profile。

不要把 `rt_subframes` 与物理步混淆：增大它只增加同一冻结状态下的渲染工作，不改变采集时间间隔。

## 8. 主要 CLI 参数分组

### 8.1 场景与相机

| 参数 | 说明 |
|---|---|
| `--usd` | 要组合打开的 USD/USDA |
| `--mapping` | 稳定语义映射 JSON |
| `--camera` | 相机绝对 Prim 路径；空值表示自动发现 |
| `--cab-root` | 驾驶室根路径 |
| `--[no-]require-camera-below-cab` | 是否要求相机是 cab 后代 |

### 8.2 采集与时间

| 参数 | 默认 | 约束 |
|---|---:|---|
| `--frames` | 50 | 正数 |
| `--width/--height` | 1280/720 | 正数 |
| `--physics-hz` | 60 | 正数 |
| `--capture-fps` | 10 | 正数且整除 physics Hz |
| `--capture-mode` | motion | static 或 motion |
| `--[no-]capture-initial-frame` | true | 是否在 t=0 采第 0 帧 |
| `--pre-roll-steps` | 0 | 非负，保持轨迹起始姿态 |

### 8.3 运动

| 参数 | 说明 |
|---|---|
| `--trajectory` | 精确五列表头的 CSV |
| `--trajectory-mode` | hold 或 loop |
| `--joint-profile` | 四关节 Articulation 契约 |
| `--articulation-ready-timeout-steps` | 等 physics tensor 的有界步数 |
| `--[no-]enable-motion` | motion 模式中是否启用关节播放 |

### 8.4 严格性与输出

| 参数 | 说明 |
|---|---|
| `--[no-]strict-mapping` | 未知语义标签是否立即中止 |
| `--[no-]strict-stage` | 通用 Stage error 是否阻断 |
| `--overwrite` | 允许写入非空输出目录 |
| `--[no-]headless` | 无窗口/有窗口运行 |

## 9. 输出目录安全

默认不允许写入非空目录。`--overwrite` 只是放行，并不会先清空旧文件。

这意味着：若原目录有 50 帧，你用 `--frames 3 --overwrite` 重跑，旧的 0003～0049 可能仍在，验证器会因文件数量超过 3 而失败。最安全做法是每次生产使用新目录，例如：

```text
output/20260721_realtime_motion_run001
output/20260721_pathtracing_static_run001
```

若确实复用目录，应由操作者先确认并清理精确目标；不要在脚本中对宽泛路径做递归删除。

## 10. 画质比较

严格匹配 Stage、相机、帧号、分辨率后，可以比较两张 RGB：

```bash
/root/isaacsim/python.sh compare_render_quality.py \
  --reference output/pathtracing_run_001/rgb/rgb_0000.png \
  --candidate output/motion_realtime_run_001/rgb/rgb_0000.png \
  --roi 100,100,800,500 \
  --output-report output/render_comparison.json
```

输出包含 MAE、RMSE、PSNR、全局 SSIM、平均亮度、近黑比例和 Laplacian 方差。

工具会把 `comparability.verified` 固定为 false，因为单凭两张像素图无法证明相机、Stage、曝光和时刻相同。应先人工或用两边 `run_config.json`、frame metadata 核对可比条件。

## 11. 一次正式生产的推荐流程

1. 跑普通 Python 单元测试；
2. 为本次运行创建唯一输出目录名；
3. 远端确认 Isaac Python 与 subLayer 路径；
4. 用 2～3 帧低分辨率 smoke test；
5. 检查 `run_config.json` 的 preflight、render effective、binding；
6. 执行 smoke 输出验证器；
7. 用正式分辨率和帧数运行到新目录；
8. 执行完整输出验证器；
9. 归档输出内的 mapping、manifest、trajectory/profile 哈希；
10. 只有验证 PASS 后才把数据交给训练流程。
